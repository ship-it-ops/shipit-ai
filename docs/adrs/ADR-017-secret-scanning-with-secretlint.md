# ADR-017: Secret Scanning via Secretlint (Pre-Commit + CI)

## Status

Accepted

## Date

2026-05-20

## Context

ShipIt-AI's repository contains config files that interpolate secrets (`${NEO4J_PASSWORD}` and friends per ADR-014), connector integration code that handles API tokens, and a `.env.example` that contributors copy into a real `.env`. The shape of the work means a careless paste can put a token, an OpenAI key, a Slack webhook, or a private key into a committed file. We need active scanning, not vigilance.

The original setup (commit `d191984`, "Add security precommit hook") wired in two scanners:

1. **Secretlint** — pre-commit via husky + lint-staged, plus a `Lint` job in CI that runs `pnpm exec secretlint --secretlintrc .secretlintrc.json "**/*"` over the full repo.
2. **gitleaks-action@v2** — a dedicated `secrets-scan` CI job using `gitleaks/gitleaks-action@v2`.

The `gitleaks-action@v2` GitHub Action requires a **paid GitGuardian / gitleaks-action license** when used on **org-owned private repositories**. ShipIt-AI is an org-owned private repo. The job started failing CI with a licensing-prompt failure on PRs, and unblocking it would require us to either pay or self-host.

The gitleaks **CLI itself** is MIT-licensed and free; only the official GitHub Action wrapper is license-gated. So we had three paths: pay, self-host the CLI in a custom CI step, or rely on Secretlint alone. Given Secretlint is already wired in two places (pre-commit and CI lint), the right move is to drop the gitleaks-action job and lean on Secretlint as the single scanner.

## Decision

Secret scanning is performed by Secretlint in two layers:

**Layer 1 — pre-commit (developer workstation).** `lint-staged` in `package.json` runs `secretlint --maskSecrets` against every staged file on every commit:

```jsonc
"lint-staged": {
  "*.{ts,tsx,js,jsx,json,md,mdx,css,yml,yaml}": "prettier --write",
  "*": "secretlint --maskSecrets"
}
```

A husky `pre-commit` hook (`.husky/pre-commit`) invokes `lint-staged`. A staged file containing what Secretlint recognizes as a secret blocks the commit, with the offending span masked in the error message. This catches almost everything before it leaves the developer's machine.

**Layer 2 — CI (full-repo backstop).** The `Lint` job in `.github/workflows/ci.yml` runs `pnpm exec secretlint --secretlintrc .secretlintrc.json "**/*"` over the whole tree. Any secret that slipped past pre-commit (e.g., `--no-verify`, a different machine, an autogenerated file) fails CI before merge.

**Rule set.** `.secretlintrc.json` uses `@secretlint/secretlint-rule-preset-recommend` — the maintainer-curated bundle covering AWS keys, GCP service accounts, GitHub tokens, JWTs, npm tokens, OpenAI keys, private keys (PEM-style), Slack tokens / webhooks, and HTTP basic-auth URLs. We do not maintain a bespoke rule set.

**Ignore list.** `.secretlintignore` excludes generated and packaged content (`node_modules/`, `dist/`, `.turbo/`, `.next/`, `coverage/`, `*.log`, `*.tsbuildinfo`, `pnpm-lock.yaml`). These either don't contain secrets by construction or contain hashes that trigger false positives.

**What we removed.** The dedicated `secrets-scan` CI job that invoked `gitleaks/gitleaks-action@v2` is deleted. The `claude-review` job's `needs:` array no longer references `secrets-scan`. The `.gitleaks*` config files are not present (we never had them — the action used its built-in defaults).

**What we did not do.** We did not switch to a free alternative like TruffleHog or detect-secrets. The reasoning is in Alternatives below.

## Consequences

### Positive

- **Same coverage at zero license cost.** Secretlint's recommended preset covers the same major secret families gitleaks scans for (AWS, GCP, GitHub, JWT, Slack, OpenAI, PEM keys, basic-auth URLs).
- **Two layers, defense in depth.** Pre-commit catches the common case; CI catches everything else. A scanner-free PR cannot land.
- **Pre-commit is a fast fail.** Staged-file scope means the hook runs in milliseconds per commit; users get instant feedback rather than waiting for CI.
- **One tool, one rule set, one config file.** No drift between two scanners disagreeing on what counts as a secret.
- **No paid SaaS dependency in CI.** The PR-failing licensing prompt is gone.

### Negative

- **Secretlint scans file contents, not git history.** Gitleaks' headline feature is diffing across the full git history to catch secrets that were committed earlier on a long branch. Secretlint sees only the current snapshot. **Mitigation:** Pre-commit blocking is the primary defense; CI's full-repo scan catches anything in the current tree. For history scanning we'd need either a layered tool (see "What we did not do") or a periodic audit run.
- **A user committing with `--no-verify` bypasses pre-commit.** **Mitigation:** CI catches it before merge. We can't prevent `--no-verify` without rebuilding the local toolchain in a sandbox, and the cost of that hardening exceeds the benefit for our threat model.
- **Recommended preset can have false positives.** **Mitigation:** `.secretlintignore` is the escape hatch for known-safe generated files. Per-line `// secretlint-disable` comments handle one-off cases. We have not needed any so far.

### Neutral

- The pre-commit hook adds ~50–200ms to every commit. Acceptable.
- Secretlint and Prettier both run via `lint-staged`. Both are stable; adding a third tool to that pipeline should be deliberate.

## Alternatives Considered

### Alternative 1: Pay for the gitleaks-action license

- **Pros:** Keep the existing CI shape. Git-history-aware scanning.
- **Cons:** Recurring cost for what amounts to a wrapper around an MIT-licensed CLI. Adds a SaaS dependency to the merge path.
- **Why rejected:** We get equivalent file-content coverage from Secretlint, which we already run. The git-history feature is nice-to-have, not load-bearing — pre-commit blocking handles the new-secret case at the source.

### Alternative 2: Run the gitleaks CLI directly (no action wrapper)

- **Pros:** Free. Git-history-aware. No SaaS dependency.
- **Cons:** Two scanners with two rule sets is two surfaces to maintain. Drift is inevitable. The "Secretlint or gitleaks — which alert do I trust?" question shows up on every false positive.
- **Why rejected:** Marginal benefit for substantial maintenance overhead. We can layer it later (Alternative 3) if Secretlint coverage proves insufficient.

### Alternative 3: Swap in TruffleHog

- **Pros:** Free, OSS, history-aware. Includes optional secret verification (probes whether a found token is live), which gitleaks lacks.
- **Cons:** Same "two scanners, two rule sets" maintenance overhead as Alternative 2 if we run it alongside Secretlint. Replacing Secretlint entirely loses the established pre-commit hook integration.
- **Why rejected (for now):** Layering TruffleHog on top is a reasonable future addition if we hit a Secretlint blind spot. Today, Secretlint alone is sufficient for our threat model and avoids the two-scanner cost.

### Alternative 4: detect-secrets (Yelp)

- **Pros:** Mature, free, baseline-driven workflow (you commit a snapshot of "known acceptable" findings and only new ones fail).
- **Cons:** Python-based, which adds a runtime to a JS/TS-only repo. The baseline workflow is heavier to operate than Secretlint's "fail the build on any finding" model.
- **Why rejected:** The Python dependency is a non-trivial cost in a JS-only monorepo, and the baseline workflow is more ceremony than we need.
